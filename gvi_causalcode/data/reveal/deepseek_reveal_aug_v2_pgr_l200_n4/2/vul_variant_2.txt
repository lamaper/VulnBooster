=== Assistant Turn 1 ===
**Scenario**
Parsing a JBIG2 text region segment from potentially untrusted input (e.g., a malicious PDF or image file). The function processes segment data containing region information, flags, Huffman table selections, and symbol dictionary references to decode a text region image.

**Vulnerability Type**
Integer overflow leading to heap buffer overflow.

**Vulnerability Trigger**
When `params.SBNUMINSTANCES` is set to a very large value via `jbig2_get_uint32(segment_data + offset)`, the subsequent loop in `jbig2_decode_text_region` (called later) may process an excessive number of symbol instances. This can cause an integer overflow in the calculation of the total number of symbols (`SBNUMSYMS`) or in internal allocations/offsets, leading to out-of-bounds memory writes in the `GR_stats` array or image buffer.

**Risky API or Data Flow**
- `jbig2_get_uint32(segment_data + offset)` without validation of `SBNUMINSTANCES`.
- `stats_size = params.SBRTEMPLATE ? 1 << 10 : 1 << 13;` followed by `GR_stats = jbig2_new(ctx, Jbig2ArithCx, stats_size);` — fixed size may be insufficient for a maliciously large number of instances.
- `jbig2_decode_text_region` internally uses `SBNUMINSTANCES` without bounds checks, potentially overflowing arithmetic in context allocations or symbol index calculations.

**Mutation Suggestions**
- Modify the `SBNUMINSTANCES` field to an extremely large value (e.g., near `UINT32_MAX`).
- Adjust `flags` to select non-Huffman (arithmetic) coding, forcing the path that calculates `SBSYMCODELEN` and allocates multiple arithmetic contexts.
- Tamper with `n_dicts` and `dicts` entries to point to malformed or empty symbol dictionaries, causing `SBNUMSYMS` to be miscalculated.
- Corrupt the `segment_data` length to cause `offset` to exceed `segment->data_length` during parsing of Huffman table data or refinement AT pixels.
